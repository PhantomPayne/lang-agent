# Core Principles — Version A: Safety-First

> **Generated by**: Core Principles Process (all 7 steps)
> **Participating Agents**: PM, Architect, New User, Evangelist, Chaos Agent, Spec Writer
> **Thematic Emphasis**: Safety and correctness above all else
> **Date**: 2026-03-16

## Tagline

**"If it compiles, it's correct — and you didn't need a PhD to write it."**

## Principles

### 1. Safe by Default

**Statement**: The language eliminates memory errors, data races, and null-pointer dereferences through its core design — without requiring the programmer to opt in to safety.

**Rationale**: Most languages treat safety as an afterthought — a linter rule, a flag, or a best-practice guideline. This language inverts the relationship: safe code is the only code the compiler accepts without annotation. Unsafe operations (raw pointer manipulation, unchecked casts, FFI calls) require explicit `unsafe` blocks with mandatory documentation comments, making every safety boundary auditable. The goal is not to prevent all unsafe code — it is to make unsafe code *visible* and *rare*.

**Corollary**:
- All references are non-nullable by default. Nullable types require explicit `Option<T>` wrapping.
- Raw memory access is gated behind `unsafe` blocks that must carry a `// SAFETY:` justification comment to pass the linter.
- The standard library exposes no function that can cause undefined behavior when called with valid arguments.
- FFI boundaries are treated as `unsafe` by default and require wrapper types to re-enter safe code.

**Counter-examples**:
- **C/C++**: Safety is entirely opt-in. Dereferencing a null pointer, reading freed memory, and signed integer overflow are all undefined behavior that compilers silently permit. Sanitizers (ASan, UBSan) exist but are not on by default.
- **Java**: Null references are the "billion-dollar mistake" — every reference type is implicitly nullable, leading to `NullPointerException` at runtime rather than a compile-time error.
- **Go**: Data races are possible because goroutines can share mutable state without compiler enforcement. The race detector is a runtime tool, not a compile-time guarantee.

### 2. Errors Are Values, Not Surprises

**Statement**: Every recoverable failure is represented as a value in the function's return type — never as hidden control flow that bypasses the caller.

**Rationale**: Exceptions create invisible control-flow paths. A function that declares `throws IOException` in Java tells you *something* can go wrong, but not *where* in the body it happens, and callers can silently ignore checked exceptions by wrapping them in `RuntimeException`. In this language, functions that can fail return `Result<T, E>`, making every error path visible at the call site. The `?` operator provides concise propagation without hiding the failure mode. Panics exist exclusively for unrecoverable logic violations (e.g., index-out-of-bounds on a provably-wrong access) and are *not* catchable in normal code — they abort the task or process.

**Corollary**:
- No `try/catch` syntax exists. Error handling uses pattern matching on `Result<T, E>` or the `?` propagation operator.
- Functions that panic are documented with `#[panics]` attributes, and calling a panicking function from a `#[no_panic]` context is a compile error.
- Libraries must not use panics for conditions that callers could reasonably handle (e.g., file-not-found, network timeout). This is enforced by an ecosystem-level lint.
- The `main` function returns `Result`, making top-level error handling explicit.

**Counter-examples**:
- **Python**: Any function can raise any exception at any time. Nothing in `def read_file(path: str) -> str` tells you it might raise `FileNotFoundError`, `PermissionError`, or `UnicodeDecodeError`. Callers discover error modes by crashing.
- **Java**: Checked exceptions were a good idea undermined by `RuntimeException`. `Integer.parseInt()` throws an unchecked `NumberFormatException`, so the compiler doesn't require callers to handle it — but it still crashes the program.
- **C++**: Exceptions interact poorly with destructors, move semantics, and performance guarantees. Many C++ codebases (Google, LLVM) ban exceptions entirely and use error codes, but the language provides no standard `Result` type.

### 3. Invalid States Are Unrepresentable

**Statement**: The type system encodes domain invariants so that malformed data cannot be constructed, turning runtime validation into compile-time guarantees.

**Rationale**: A `User` struct with a `String` email field allows the empty string, `"not-an-email"`, and even binary garbage. A `User` struct with an `EmailAddress` field — backed by a type whose constructor validates format — makes it impossible for downstream code to encounter an invalid email. This principle extends beyond wrapper types: sum types (tagged unions) replace boolean flags and stringly-typed enums, ensuring that pattern matches are exhaustive and that impossible states have no representation.

**Corollary**:
- Sum types (enums with data) are a first-class feature, and `match` expressions must be exhaustive — no default/wildcard arms are required if all variants are covered, but missing a variant is a compile error.
- The `newtype` pattern is zero-cost: wrapping `u32` in `UserId(u32)` adds no runtime overhead but prevents accidentally passing a `ProductId` where a `UserId` is expected.
- Boolean function parameters are discouraged by the linter in favor of two-variant enums (e.g., `Recursive::Yes | Recursive::No` instead of `true/false`).
- Struct constructors can enforce invariants (e.g., `NonEmptyList<T>` has no public way to create an empty instance).

**Counter-examples**:
- **Go**: The lack of sum types means error states are encoded as `(value, error)` pairs where *both* can be non-nil or *both* can be nil — a state that should be unrepresentable. Developers must remember to check the error, and the compiler does not enforce it.
- **JavaScript/TypeScript**: TypeScript's structural typing helps, but `any` and type assertions (`as`) provide escape hatches that bypass the type system entirely. A `userId: string` provides no domain-level protection.
- **C**: Enums are integers. Nothing prevents assigning `42` to an enum that only defines values 0–3. Flag fields use bitwise operations on `int` with no compiler-checked meaning.

### 4. Catch Bugs at Compile Time

**Statement**: The compiler is the primary line of defense — it should reject incorrect programs rather than allowing them to fail at runtime, even if this means longer compile times or more annotation burden.

**Rationale**: A bug caught by the compiler costs minutes. A bug caught in production costs hours, reputation, and sometimes safety. This language explicitly trades compile-time cost for runtime correctness. The type checker, borrow checker, and effect system work together to verify properties that other languages defer to tests or runtime checks: memory safety, thread safety, exhaustive error handling, and resource cleanup. The compiler's error messages are treated as a user interface — they must explain *why* the code is wrong, *what* the programmer likely intended, and *how* to fix it.

**Corollary**:
- Unused variables, unreachable patterns, and non-exhaustive matches are hard errors, not warnings.
- Implicit numeric conversions (e.g., `i32` to `i64`) are forbidden; all widening/narrowing requires explicit syntax.
- The compiler provides structured error messages with source spans, fix suggestions, and links to explanatory documentation (modeled after Rust's and Elm's error UX).
- Dead code and unused dependencies are compile errors in release mode to prevent bitrot.

**Counter-examples**:
- **C**: Implicit integer promotion silently converts `char` to `int`, signed to unsigned, and narrow to wide — causing subtle arithmetic bugs that are nearly impossible to spot in code review.
- **Python**: Type errors are discovered at runtime. `def add(a, b): return a + b` happily concatenates strings, adds integers, or crashes on incompatible types — all at runtime. Mypy exists but is optional and incomplete.
- **Haskell**: Gets compile-time checking right, but error messages are notoriously unhelpful — a missing typeclass instance produces a wall of text about constraint resolution that novices cannot parse. Our language treats error UX as a first-class design concern.

### 5. Progressive Disclosure of Complexity

**Statement**: A newcomer should write correct, safe programs on their first day — advanced features remain invisible until the programmer reaches for them.

**Rationale**: Rust proved that a language *can* guarantee memory safety at compile time. It also proved that requiring every programmer to understand lifetimes, trait bounds, and pin projections creates a brutal learning curve. This language layers its complexity: beginners write code that looks like Python or Go, using owned values and automatic memory management via scope-based destruction. Intermediate users learn borrowing for performance. Advanced users access lifetime annotations, custom allocators, and unsafe blocks. At no layer does the programmer encounter concepts from a higher layer unless they explicitly opt in.

**Corollary**:
- "Hello, World" requires no lifetime annotations, no trait bounds, no explicit type parameters, and no understanding of ownership — values are moved by default and the compiler handles the rest.
- Borrowing (`&T`, `&mut T`) is introduced only when the programmer needs shared or mutable access without transferring ownership. The compiler suggests borrowing when a move would be a compile error.
- Generic functions use type inference aggressively — `fn max(a, b)` infers the `Ord` constraint from usage rather than requiring `fn max<T: Ord>(a: T, b: T)` upfront.
- The standard library provides high-level APIs (e.g., `fs.read_text(path)`) alongside low-level ones (e.g., `fs.open(path, options).read_into(buffer)`), with the high-level API documented first.

**Counter-examples**:
- **Rust**: A beginner writing a linked list encounters lifetimes, `Box`, `Option`, `as_ref`, `as_mut`, and the borrow checker simultaneously. The learning cliff is real — surveys consistently cite it as the top barrier to adoption.
- **C++**: Template metaprogramming, SFINAE, and concepts are required for writing generic library code, but they leak into error messages for *users* of those libraries. A typo in a `std::vector` operation can produce hundreds of lines of template errors.
- **Haskell**: Monad transformers, higher-kinded types, and type families are often required for practical I/O and error handling — there is no "simple subset" that avoids them for real programs.

### 6. Clarity at the Call Site

**Statement**: Reading a function call should reveal its failure modes, side effects, and ownership transfers — code must be locally understandable without navigating to definitions.

**Rationale**: Code is read far more often than it is written. A function call like `process(data)` tells the reader nothing: does it consume `data`? Can it fail? Does it perform I/O? In this language, the call site makes these properties visible. Moves are syntactically distinct from borrows. Fallible functions return `Result` and require `?` or explicit matching. Functions with side effects carry an effect annotation that is visible in the signature. The goal is that a code reviewer can assess correctness by reading the caller, without opening the callee.

**Corollary**:
- Passing ownership uses `move data` or bare `data` (moved by default); borrowing uses `&data` or `&mut data`. The distinction is visible at every call site.
- Functions performing I/O, network access, or filesystem operations carry an `effect` marker (e.g., `fn read_config(path: Path) -> Result<Config, IoError> with io`) that is visible in documentation and enforced by the compiler.
- Method chaining must preserve readability — the linter enforces a maximum chain length and requires intermediate bindings for complex transformations.
- No operator overloading for domain types by default. `a + b` always means numeric addition unless the type explicitly implements `Add`, and the implementation must be associative and commutative per the trait contract.

**Counter-examples**:
- **Kotlin**: Extension functions, scope functions (`let`, `run`, `apply`, `also`, `with`), and operator overloading make call sites ambiguous. `data.let { process(it) }` obscures whether `data` is consumed, borrowed, or copied.
- **Scala**: Implicit conversions and implicit parameters mean that `process(data)` may silently convert `data` to a different type or inject parameters that are invisible at the call site. Scala 3 improved this but the legacy remains.
- **Ruby**: Method calls, blocks, `method_missing`, and open classes mean that `user.save!` could do literally anything — the reader cannot know without checking the runtime class, every included module, and every monkey-patch loaded by the application.

### 7. Predictable Performance Cost

**Statement**: Every abstraction has a knowable, documentable performance cost — the language avoids hidden allocations, implicit copies, and opaque runtime magic.

**Rationale**: "Zero-cost abstractions" is a noble goal but an imprecise promise. Some safety features *do* impose cost — bounds checking on array access, reference counting for shared ownership, stack-unwinding for panics. This language does not pretend these are free. Instead, it makes costs *predictable* and *visible*: the documentation for every standard library type states its allocation behavior, every implicit copy is flagged by the linter, and the compiler can emit a cost annotation showing where heap allocations occur in a function. Developers can reason about performance without profiling.

**Corollary**:
- Types are `Copy` only if they are trivially copyable (stack-allocated, no heap indirection). All other types are moved by default, and cloning requires an explicit `.clone()` call.
- The standard library documents allocation behavior for every method: `Vec::push` documents its amortized O(1) cost and its worst-case reallocation. `HashMap::insert` documents its load-factor threshold.
- A `#[no_alloc]` function attribute forbids heap allocation within the function body, enforced by the compiler — useful for real-time or embedded contexts.
- The compiler provides an opt-in "cost model" view that annotates source lines with estimated stack usage, heap allocations, and branch misprediction hints.

**Counter-examples**:
- **Java**: Autoboxing silently converts `int` to `Integer`, allocating on the heap. `String` concatenation in a loop silently creates O(n) intermediate objects. The JIT compiler *may* optimize these away — or may not. Performance is unpredictable without a profiler.
- **Python**: Every integer is a heap-allocated object. Every attribute access is a dictionary lookup. Every function call allocates a frame object. The cost model is "everything is expensive" but nothing makes this visible.
- **Go**: Escape analysis decides whether a variable is stack- or heap-allocated, but the decision is invisible to the programmer and can change between compiler versions. A seemingly innocent closure can cause an unexpected heap allocation.

## Process Notes

### Step 1: Seed Discussion Summary

The **PM** framed the problem: existing languages force a choice between safety (Rust, Haskell) and accessibility (Go, Python). The target audience is professional developers building reliable services, CLI tools, and data systems who want compile-time correctness without Rust's learning curve.

The **Architect** proposed 10 candidate principles: (1) Safe by Default, (2) Errors Are Values, (3) Make Illegal States Unrepresentable, (4) Compile-Time Over Run-Time, (5) Ownership Without Obscurity, (6) Progressive Disclosure of Complexity, (7) Explicit Over Implicit, (8) One Obvious Way, (9) Fearless Concurrency, (10) Zero-Cost Safety. These drew from Rust's ownership model, Haskell's type-driven design, Elm's error messages, Go's simplicity, and Swift's optional handling.

### Step 2: Chaos Review Summary

The **Chaos Agent** challenged each candidate:

- *Safe by Default*: FFI is inherently unsafe — if interop is too painful, adoption suffers. Resolved by making unsafe explicit, not forbidden.
- *Errors Are Values*: What about truly unrecoverable errors (OOM, stack overflow)? Resolved by distinguishing panics (unrecoverable) from Results (recoverable), and making panics non-catchable.
- *Illegal States*: Encoding every invariant in types leads to Haskell-style type-level programming that's incomprehensible. Resolved by focusing on sum types and newtypes, not dependent types.
- *Compile-Time Over Run-Time*: Excessive static checking causes slow builds (Rust, Scala). Resolved by reframing as "catch bugs early" with a commitment to error-message quality.
- *Zero-Cost Safety*: Bounds checks *do* cost. "Zero-cost" is misleading. Resolved by renaming to "Predictable Performance Cost."
- *Explicit Over Implicit*: Go's `if err != nil` proves over-explicitness kills ergonomics. Resolved by merging into "Clarity at the Call Site" — visible where it matters, inferred where it doesn't.
- *One Obvious Way*: Different domains need different patterns. Dropped as a standalone principle; subsumed into standard library design guidelines.
- *Fearless Concurrency*: Not all programs need concurrency; encoding thread-safety everywhere taxes single-threaded code. Subsumed into "Safe by Default" — data races are prevented by the same ownership system, not a separate mechanism.

### Step 3: User Lens Summary

The **New User** prioritized: Safe by Default ("protect me from myself"), Errors Are Values ("I hate debugging surprise exceptions"), and Progressive Disclosure ("don't make me learn everything on day one"). They flagged "Ownership Without Obscurity" as jargon ("I don't know what ownership means yet") and "Zero-Cost Safety" as an implementer concern. They requested a readability commitment ("I want to read other people's code") — addressed by adding "Clarity at the Call Site." They also asked: "Can I be productive in a week?" — directly addressed by Progressive Disclosure's corollaries.

### Step 4: Evangelist Lens Summary

The **Evangelist** identified the core narrative: "Rust proved safety is possible but demanded too much. We prove safety can be *accessible*." The killer combination is Safe by Default + Progressive Disclosure — these together answer "Why not just use Rust?" The tagline — *"If it compiles, it's correct — and you didn't need a PhD to write it"* — anchors the pitch. "Catch Bugs at Compile Time" is the technical differentiator against Go and Python. "Predictable Performance Cost" differentiates against Java and managed-language overhead. "One Obvious Way" was flagged as a Python echo and dropped as a standalone principle. "Fearless Concurrency" was flagged as a Rust echo and merged.

### Step 5: Synthesis Summary

The **Architect** and **Spec Writer** consolidated 10 candidates into 7 final principles through three operations:

1. **Merge**: "Zero-Cost Safety" → "Predictable Performance Cost" (per Chaos Agent). "Explicit Over Implicit" → "Clarity at the Call Site" (per New User). "Fearless Concurrency" and "Ownership Without Obscurity" → subsumed into "Safe by Default" and "Progressive Disclosure."
2. **Drop**: "One Obvious Way" demoted to a standard library design guideline, not a language principle.
3. **Refine**: "Compile-Time Over Run-Time" became "Catch Bugs at Compile Time" with an explicit commitment to error-message quality (per Chaos Agent's concern about slow builds, and Haskell's counter-example of bad errors). "Make Illegal States Unrepresentable" → "Invalid States Are Unrepresentable" with focus on practical type features (sum types, newtypes) rather than dependent-type theory.

Each principle received corollaries grounding abstract statements in concrete language decisions, and counter-examples referencing specific failures in C, C++, Java, Python, Go, Haskell, Kotlin, Scala, and Ruby.

### Step 6: PM Review

The **PM** conducted a final review:

- **Redundancy check**: Principles 1 (Safe by Default) and 4 (Catch Bugs at Compile Time) overlap in spirit but differ in scope — #1 defines the *posture* (safety is default), #4 defines the *mechanism* (the compiler is the enforcer). Kept as separate principles.  Principle 3 (Invalid States) could be a sub-principle of #1, but it makes a specific *type-system* commitment that #1 does not, so it stands alone.
- **Contradiction check**: Principle 5 (Progressive Disclosure) could conflict with #6 (Clarity at the Call Site) — if we hide complexity, how is everything visible? Resolution: Progressive Disclosure controls *what syntax a beginner must learn*, not *what information is available*. Full type signatures, effect annotations, and ownership markers are always present in the compiled output and IDE — they're just inferred rather than written.
- **Memorability check**: Seven principles, each with a distinct verb/noun anchor: Safe, Errors, Invalid States, Catch, Progressive, Clarity, Predictable. A developer can recall the list.
- **Consensus vote**: All six agents affirm. The Chaos Agent notes that the corollaries adequately address edge cases. The New User confirms the language "sounds like something I'd try." The Evangelist confirms the narrative holds together.

**Result**: Consensus reached. Seven principles committed as Version A (Safety-First).
