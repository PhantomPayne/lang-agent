# Core Principles — Version C: Innovation-First

> **Generated by**: Core Principles Process (all 7 steps)
> **Participating Agents**: PM, Architect, New User, Evangelist, Chaos Agent, Spec Writer
> **Thematic Emphasis**: Expressiveness and innovation above all else
> **Date**: 2026-03-16

## Tagline

**"The language that refuses to limit what you can express."**

## Principles

### 1. The Program Is the Programmable

**Statement**: Every construct the language provides — from control flow to type declarations to module systems — is a programmable surface that user code can inspect, generate, and transform.

**Rationale**: The deepest source of power in Lisp-family languages is that programs are data structures the programmer can manipulate. But Lisp achieves this by flattening syntax into S-expressions, sacrificing readability for uniformity. We take a different path: rich, structured syntax backed by a first-class AST that macros and compiler plugins operate on as typed data. If the language uses a construct internally, that construct must be available as an API to user code — no privileged builtins that only the compiler can emit.

**Corollary**:
- The AST is a documented, versioned data type. Macros receive and return typed AST nodes, not opaque token streams.
- `if`, `match`, `for`, and `fn` are not keywords with hardcoded behavior — they are desugaring targets that user-defined macros can emit, wrap, or replace.
- The module system, trait resolution, and monomorphization passes are exposed as compiler-extension points with stable interfaces.
- A "language-level feature" and a "library-contributed feature" must be indistinguishable at the call site.

**Counter-examples**:
- **Rust**: Procedural macros operate on `TokenStream`, an untyped sequence of tokens, making it difficult to perform semantic transformations or provide meaningful error messages during macro expansion.
- **Go**: The language has no macro system or metaprogramming beyond `go generate`, an external code-generation tool that cannot participate in type checking or interact with compiler internals.
- **Java**: Annotation processors can generate new classes but cannot modify existing ones, and language constructs like `synchronized` or `instanceof` pattern matching are sealed compiler magic unavailable to user-defined abstractions.

### 2. Compile-Time Is the First Runtime

**Statement**: The compile-time environment is a full execution environment — the same language that runs at runtime runs at compile time, with access to the type system, file system, and network.

**Rationale**: Most languages treat compilation as a limited, special phase where only a restricted subset of operations are permitted. C++ `constexpr` can evaluate a growing but still limited set of expressions; Rust's `const fn` excludes heap allocation and trait objects. We reject this artificial boundary. If the programmer can write a function that runs at runtime, the same function — without annotation or restriction — should be callable at compile time. The compiler is a program; compilation is execution; the phase distinction is a spectrum, not a wall.

**Corollary**:
- Any function is callable at compile time unless it performs I/O that the build system has not granted permission for (compile-time I/O is capability-gated, not prohibited).
- Generic parameters can be computed values, types, or functions — not just types as in most languages.
- Compile-time evaluation can query the type system: "does type T implement trait X?" is a compile-time boolean, not a SFINAE hack.
- Code generation, conditional compilation, and build configuration are written in the same language, not in a preprocessor dialect or build DSL.

**Counter-examples**:
- **C++**: `constexpr` and `consteval` impose a growing but still restrictive subset of C++; template metaprogramming is a separate, accidentally-Turing-complete language with incomprehensible error messages.
- **Zig**: Comes closest to this principle with `comptime`, but compile-time code cannot perform I/O or interact with external systems, limiting its use for build-time code generation from schemas or network resources.
- **Rust**: `const fn` is a restricted subset that excludes traits, heap allocation, and most of the standard library, forcing macro authors to operate at the token level instead of the semantic level.

### 3. Types Serve Expression, Not Bureaucracy

**Statement**: The type system exists to make new abstractions possible and safe — not to require the programmer to satisfy the compiler with annotations it could infer.

**Rationale**: A powerful type system is one of the greatest tools for expressiveness: algebraic data types let you model domains precisely, higher-kinded types let you abstract over containers, and dependent types let you encode invariants as types. But type systems often devolve into paperwork — verbose annotations, ceremony around trivial conversions, and arcane error messages when inference fails. We commit to a type system that is powerful in what it can *express* and minimal in what it *demands*. Inference should be aggressive, error messages should speak in the programmer's domain language, and the type system should feel like a creative partner, not a bureaucratic gatekeeper.

**Corollary**:
- Top-level function signatures are inferred in non-public APIs; public API boundaries require explicit signatures as documentation.
- Higher-kinded types, GADTs, and row polymorphism are available but do not require the programmer to name them when usage is unambiguous.
- Type errors reference the programmer's source code and domain concepts, not the compiler's internal constraint-solving steps.
- Subtyping, if present, has a simple and predictable variance model — no wildcard captures, no existential types leaking into user-facing signatures.

**Counter-examples**:
- **Haskell**: Powerful type system, but GHC error messages routinely expose internal constraint-solver state, and advanced features (`TypeFamilies`, `GADTs`, `DataKinds`) require enabling language extensions that fragment the ecosystem.
- **Java**: Generics are both less powerful (type erasure, no higher-kinded types) and more verbose (wildcard captures, bounded type parameters) than necessary; the type system adds ceremony without proportional expressiveness.
- **TypeScript**: Structural typing and inference are strong, but the type system's interaction with JavaScript semantics creates unsound corners (`any` propagation, unsound covariance of arrays) that undermine trust.

### 4. Paradigms Are Tools, Not Religions

**Statement**: The language provides first-class support for functional, imperative, and concurrent programming — the programmer selects the paradigm that fits the problem, and the language does not punish the choice.

**Rationale**: Real programs are not purely functional or purely imperative. A web server's request handler might parse JSON functionally, update a database imperatively, and fan out notifications concurrently. Languages that privilege one paradigm force programmers into unnatural contortions when the problem doesn't fit: Haskell's IO monad is a brilliant solution to a problem that imperative languages don't have; Go's lack of generics (until 1.18) forced copy-paste where parametric polymorphism was the natural solution. We support all three major paradigms with equal ergonomics, no monad tax for side effects, no ceremony for immutability, and structured concurrency as a first-class concern rather than a library bolt-on.

**Corollary**:
- Immutable bindings are the default, but mutable bindings require only a single keyword — not a type-system encoding.
- Pattern matching, algebraic types, and closures work seamlessly alongside mutable state, loops, and imperative I/O.
- Side effects are tracked in the type system *optionally* — purity can be enforced per-module, not globally.
- Structured concurrency (task groups, scoped spawning, cancellation) is built into the language, not a library pattern.

**Counter-examples**:
- **Haskell**: Purity is enforced globally via the IO monad, making simple imperative tasks (reading a file, printing debug output) require monadic plumbing that obscures intent for non-functional programmers.
- **Go**: Imperative-first design with goroutines but no algebraic data types, no pattern matching (until recently, limited), no sum types — functional approaches are second-class or impossible.
- **Scala**: Supports multiple paradigms but the interaction between object-oriented inheritance, implicit resolution, and functional patterns creates a combinatorial complexity that few programmers can fully model.

### 5. Orthogonal by Design

**Statement**: Language features are small, independent, and freely composable — a feature's behavior does not change based on which other features are active.

**Rationale**: The most expressive languages are not those with the most features, but those whose features compose without interference. When features interact in surprising ways — C++'s interaction between templates, overload resolution, and implicit conversions; Python's interaction between decorators, metaclasses, and multiple inheritance — the resulting complexity is multiplicative, not additive. We commit to a feature design discipline: each feature does one thing, interacts with other features through documented composition rules, and does not alter its semantics based on context. Two features that cannot compose cleanly should not both exist.

**Corollary**:
- Generics, traits, and effects compose: a function can be generic over a type, constrained by a trait, and annotated with an effect, and the meaning is the intersection of each feature's individual meaning.
- There is one mechanism for polymorphism (traits/typeclasses), not three (inheritance, interfaces, duck typing).
- No feature has "special cases when combined with feature X" — if such a case arises, the features are redesigned until they compose cleanly or one is removed.
- Error handling, resource management, and concurrency use composable, algebraic structures (sum types, linear types, effect handlers) rather than ad-hoc language mechanisms (`try/catch/finally`, `defer`, `async/await` coloring).

**Counter-examples**:
- **C++**: Templates, overload resolution, implicit conversions, SFINAE, concepts, and modules interact in ways that no single person fully understands; the language is a product of accretion, not composition.
- **Python**: Metaclasses, `__init_subclass__`, decorators, descriptors, and `__slots__` all modify class behavior through overlapping, sometimes conflicting mechanisms.
- **Rust**: `async fn` introduces a function-coloring problem — async and sync code use different traits (`AsyncRead` vs. `Read`), different iteration patterns, and different error-handling idioms, violating composability.

### 6. Power Must Be Legible

**Statement**: Every abstraction, no matter how powerful, must produce code that a competent programmer can read, trace, and debug without special tooling.

**Rationale**: Metaprogramming and advanced type systems create a tension: the author gains enormous power, but readers — including the author six months later — may face code that is effectively obfuscated. Lisp macros can rewrite control flow invisibly; Haskell's typeclass resolution can insert function calls that appear nowhere in the source; C++ templates can generate thousands of lines of code from a single declaration. We resolve this tension by requiring legibility as a design constraint on every power feature. Macros must be syntactically visible at the call site. Type-level computation must be inspectable. Generated code must be viewable. Power without legibility is not expressiveness — it is obscurantism.

**Corollary**:
- Macros are invoked with a distinct syntax (e.g., `!` suffix or `#[]` attribute) that is always visible in source — no "transparent" macros that look like function calls but rewrite control flow.
- The compiler provides a `--expand` mode that shows the fully expanded, desugared code for any expression, function, or module.
- Trait/typeclass resolution is deterministic and inspectable: the compiler can report exactly which implementation was selected and why.
- Implicit conversions are either absent or limited to a small, closed set of safe coercions (numeric widening, reference reborrowing) — never user-defined implicit conversions.

**Counter-examples**:
- **Scala 2**: Implicit conversions and implicit parameters could insert arbitrary code at any point, making it impossible to understand what a line of code does without running the full implicit resolution algorithm in your head.
- **C++ (templates)**: Template instantiation errors produce pages of output referencing internal implementation details of the standard library, not the programmer's code.
- **Elixir**: Macros are powerful and heavily used (the `if` construct is itself a macro), but macro-generated code is invisible in the source and can only be inspected via `Macro.expand/2` in an interactive session.

### 7. Complexity Is Earned, Not Inherited

**Statement**: Every unit of complexity in the language must justify itself by enabling a form of expression that would otherwise be impossible or unsafe — complexity inherited from historical accident or backward compatibility is refused.

**Rationale**: C++ is the cautionary tale: every feature is individually justified, but the aggregate complexity is crushing because each feature must coexist with 40 years of accumulated decisions. We start from the position that the language's complexity budget is finite and precious. A feature is admitted only if it enables a genuinely new capability — not merely a syntactic convenience for an existing one. When two designs offer similar expressiveness, we choose the simpler one even if the more complex design is marginally more powerful. And we refuse to carry complexity forward from prior languages simply because programmers are accustomed to it: familiarity is not a justification for a bad design.

**Corollary**:
- No null: option types encode absence. No exceptions as control flow: sum types encode failure. No header files: modules encode visibility.
- A feature proposal must demonstrate a program that is *impossible* to write without it, not merely *inconvenient*.
- Backward compatibility with prior versions does not justify retaining a feature that violates other principles — migration tooling and edition mechanisms are used instead.
- The language has a formal complexity budget process: adding a feature requires either removing a feature of equivalent complexity or demonstrating that the new feature simplifies existing patterns enough to offset its cost.

**Counter-examples**:
- **C++**: Retains `NULL`, raw pointers, C-style casts, `#include`, and trigraphs alongside their modern replacements, because backward compatibility is treated as inviolable.
- **Python**: Carries both `%`-formatting, `str.format()`, and f-strings; the GIL; and a class system that evolved from old-style to new-style classes — complexity inherited rather than earned.
- **JavaScript**: `var` vs. `let` vs. `const`, `==` vs. `===`, prototype chains alongside `class` syntax, and `this` binding rules are all complexity inherited from hasty early design decisions that can never be removed.

## Process Notes

### Step 1: Seed Discussion Summary

The **PM** framed the design space: a language for advanced programmers who are frustrated by the expressiveness ceiling of mainstream languages but unwilling to accept the usability tradeoffs of research languages. The target sits between Rust (systems power, steep learning curve, limited metaprogramming) and Haskell (type-level power, poor tooling story for industry, monadic ceremony). The **Architect** proposed 10 candidate principles:

1. Code is Data, Data is Code
2. Types as a Creative Medium
3. Compile-Time is the First Runtime
4. No Artificial Ceilings
5. Composition Over Accumulation
6. Semantic Transparency
7. Paradigm Fluidity
8. Controlled Unsafety
9. Expressiveness Justifies Complexity
10. The Language Evolves With Its Users

### Step 2: Chaos Review Summary

The **Chaos Agent** challenged each principle with adversarial scenarios:

- **"Code is Data"** invites macro-generated security vulnerabilities invisible in source review — proposed inversion: *"Keep code and data separate for auditability."*
- **"Compile-Time is the First Runtime"** risks C++-style compile times that destroy developer feedback loops — proposed inversion: *"Fast feedback matters more than compile-time power."*
- **"Types as Creative Medium"** risks Scala-style type astronautics where only PhD students can read library signatures — proposed inversion: *"Types serve the value level, not the other way around."*
- **"Paradigm Fluidity"** risks an "everything bagel" language with no coherent identity — proposed inversion: *"One paradigm, done perfectly."*
- **"No Artificial Ceilings"** risks users coupling to unstable compiler internals, breaking on every release — proposed inversion: *"Stable abstractions over exposed internals."*
- **"Expressiveness Justifies Complexity"** is exactly the argument that turned C++ into what it is — proposed inversion: *"Simplicity is the ultimate expressiveness."*
- **"Controlled Unsafety"** tends to become the default when the safe path is too restrictive — proposed inversion: *"Safe by default, no escape needed."*

The Chaos Agent's challenges were instrumental in adding the legibility principle and the complexity-budget mechanism.

### Step 3: User Lens Summary

The **New User** responded:

- *"Compile-Time is the First Runtime"* and *"No Artificial Ceilings"* immediately signal that this language will respect my intelligence — these are compelling.
- *"Code is Data"* sounds like a Lisp pitch, not a user benefit. Reframe around what I can *do* with it: extend the language, build DSLs, generate boilerplate.
- *"Semantic Transparency"* is the right idea but the name is opaque. Tell me I can *read and debug* my code — that's what I care about.
- **Missing**: What about error messages? If the type system is this powerful, I need confidence that the compiler will *help* me, not drown me in constraint-solver output.
- **Missing**: What about simplicity of simple things? Can I write a script without understanding higher-kinded types?

These concerns led to the "Types Serve Expression, Not Bureaucracy" framing and the legibility-focused error message corollaries.

### Step 4: Evangelist Lens Summary

The **Evangelist** identified narrative anchors:

- **Strong hooks**: "Compile-Time is the First Runtime" (unique, memorable, demonstrable), "The Program is the Programmable" (captures the metaprogramming vision without Lisp jargon), "Complexity is Earned" (positions against C++ directly).
- **Too vague**: "Composition Over Accumulation" — every language claims composability. Reframed as "Orthogonal by Design" with the concrete commitment that features must not change behavior when combined.
- **Tagline**: After iterating through candidates ("Think it, type it, the compiler does the rest" — too passive; "Every abstraction you need, no abstraction you can't change" — too long), settled on: **"The language that refuses to limit what you can express."**

### Step 5: Synthesis Summary

The **Architect** and **Spec Writer** merged and refined the 10 candidates into 7 principles:

- "Code is Data" + "No Artificial Ceilings" → **The Program Is the Programmable** (addresses user feedback about framing)
- "Compile-Time is the First Runtime" → retained with capability-gated I/O to address the Chaos Agent's concern
- "Types as Creative Medium" → **Types Serve Expression, Not Bureaucracy** (addresses user concern about type astronautics, adds error message commitments)
- "Paradigm Fluidity" → **Paradigms Are Tools, Not Religions** (sharper framing, addresses "everything bagel" concern by committing to equal ergonomics rather than maximal features)
- "Composition Over Accumulation" → **Orthogonal by Design** (concrete composition rules, addresses Evangelist's vagueness critique)
- "Semantic Transparency" → **Power Must Be Legible** (merges with Chaos Agent's auditability concern and User's readability concern)
- "Expressiveness Justifies Complexity" → **Complexity Is Earned, Not Inherited** (inverts the burden of proof, adds formal complexity budget)
- "Controlled Unsafety" was dropped as a standalone principle — its concerns are subsumed by "Power Must Be Legible" and "Orthogonal by Design."
- "The Language Evolves With Its Users" was dropped — it is an ecosystem principle, not a language-design principle, and is better addressed in a governance document.

### Step 6: PM Review

The **PM** conducted a final review:

- **Redundancy check**: Principles 1 and 2 (Programmable + Compile-Time) are related but distinct — 1 is about *what* you can manipulate, 2 is about *when* you can execute. No merge needed.
- **Contradiction check**: Principle 4 (Paradigms) and Principle 5 (Orthogonal) could tension — supporting many paradigms might challenge orthogonality. Resolution: the paradigms themselves are the orthogonal axes; the commitment is that they compose, not that they multiply.
- **Memorability check**: All 7 principles have distinct, memorable names. The list passes the "can you recite them from memory after one read" test.
- **Consensus vote**: PM calls vote. Architect: ✅. New User: ✅ (with note that error messages must be a launch priority). Evangelist: ✅. Chaos Agent: ✅ (with note that the complexity budget must be a real process, not aspirational). Spec Writer: ✅.

**Consensus reached unanimously.** The Chaos Agent's and New User's conditions are recorded as implementation commitments, not principle amendments.
