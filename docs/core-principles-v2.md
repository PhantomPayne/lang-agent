# Core Principles — Version B: Ergonomics-First

> **Generated by**: Core Principles Process (all 7 steps)
> **Participating Agents**: PM, Architect, New User, Evangelist, Chaos Agent, Spec Writer
> **Thematic Emphasis**: Developer experience and ease of use above all else
> **Date**: 2026-03-16

## Tagline

**"Simple to start, powerful to stay."**

## Principles

### 1. Readable at a Glance

**Statement**: Code should communicate intent to a human reader before it communicates instructions to a machine.

**Rationale**: Most code is read far more often than it is written. A language optimized for authoring speed at the expense of reading speed creates a net productivity loss across a codebase's lifetime. Readability is not subjective decoration — it is a measurable property: can a developer unfamiliar with this module correctly predict its behavior by reading it once, top to bottom?

**Corollary**:
- Minimal use of sigils and symbolic operators beyond arithmetic and comparison. No `::`, `->`, `=>`, `<>` where a keyword or method call would suffice.
- Control flow uses keywords (`if`, `for`, `match`), not overloaded operators.
- Naming conventions are enforced by the formatter: `snake_case` for values, `PascalCase` for types. No debate, no style guides to maintain.
- String interpolation uses a single, obvious syntax — not three competing mechanisms.

**Counter-examples**:
- **Rust**: Lifetime annotations (`'a`), turbofish (`::<>`), and trait bounds syntax (`where T: Into<String> + Clone`) are powerful but create a wall of symbols that new readers cannot parse without training.
- **Perl**: The motto "There's More Than One Way To Do It" produces code where readability depends entirely on the author's stylistic preferences.
- **Haskell**: Operator sections (`(+3)`) and point-free style (`map . filter`) are elegant to experts but opaque to anyone reading the code for the first time.
- **C++**: Overloaded `<<` for stream output means `std::cout << x << std::endl` reads as bit-shifting to anyone from another language.

### 2. One Obvious Path

**Statement**: For any common task, there should be one clear, idiomatic way to accomplish it, and the language should make that path the easiest one to find.

**Rationale**: Decision fatigue is a real cost. When a developer must choose between three HTTP libraries, two string formatting approaches, or four concurrency models before writing their first useful line, the language has failed at its job. "One obvious path" does not mean restricting power — it means curating defaults so that the 90% case requires zero research.

**Corollary**:
- The standard library provides exactly one blessed approach for: HTTP, JSON, file I/O, testing, logging, and CLI argument parsing.
- Concurrency has a single default model (structured concurrency with async/await). Lower-level primitives exist but are in an explicitly-named `unsafe` or `low_level` module.
- String formatting uses one interpolation syntax. Period.
- The official tutorial never says "you could also…" — it says "do this."

**Counter-examples**:
- **JavaScript/Node.js**: Callbacks, Promises, and async/await coexist. A new developer must understand all three to read existing code, and library authors must support all three.
- **Python**: `%` formatting, `.format()`, and f-strings are all "Pythonic." A function might use all three in one file. The `urllib`/`urllib2`/`requests` split forces every new user to Google "which HTTP library should I use?"
- **Scala**: Implicits, type classes, inheritance, and mixins all solve overlapping problems. Two idiomatic Scala codebases can look like different languages.
- **C++**: Smart pointers (`unique_ptr`, `shared_ptr`, `weak_ptr`), raw pointers, and references all manage memory with overlapping use cases and subtle tradeoffs that even experienced developers debate.

### 3. Errors That Teach

**Statement**: Every error message is a user interface. It must locate the problem, explain why it is wrong, and suggest a concrete fix.

**Rationale**: The error message is the most-read output of any compiler or runtime. A cryptic error wastes minutes per occurrence, multiplied across every developer, every day. Investing in error quality has a higher return-on-effort than almost any language feature. This principle is a design *constraint*: if a language feature cannot produce a clear error message when misused, the feature must be redesigned until it can.

**Corollary**:
- Compiler errors include: the exact source location (with a code snippet and underline), a plain-English explanation, and a suggested fix with corrected code when possible.
- Error codes are stable and searchable. Every error has a unique code (e.g., `E0312`) that links to an extended explanation with examples.
- Runtime errors include a full stack trace with local variable values at each frame, not just function names.
- Type mismatch errors name both the expected and actual types and show where the expectation originated (e.g., "expected `String` because of the return type declared on line 14").
- The compiler never says just "syntax error" or "type mismatch." Those are bugs in the compiler.

**Counter-examples**:
- **C++**: Template errors in GCC/Clang routinely produce 200+ lines of nested type expansion that even experts cannot parse. `std::vector<std::map<int, std::string>>` errors expose the full template machinery.
- **Java**: `NullPointerException` with no indication of *which* reference was null (improved only in Java 14, decades after launch).
- **TypeScript**: Complex generic errors like `Type 'X' is not assignable to type 'Y'` with deeply nested conditional types produce error messages longer than the source file.
- **Haskell**: "No instance for (Show (IO ()))" tells an expert everything and a beginner nothing.

### 4. Progressive Disclosure

**Statement**: A beginner's first program should require no concepts beyond the immediate task; advanced features exist but never leak into simple code.

**Rationale**: The steepest part of a learning curve is the first hour. If "Hello, World" requires understanding modules, types, entry points, or build systems, the language has front-loaded its complexity budget on the people least equipped to handle it. Progressive disclosure means that features are layered: you encounter generics only when you need a generic container, you encounter concurrency only when you need parallelism, and the syntax for simple code is never polluted by the existence of advanced features.

**Corollary**:
- "Hello, World" is one line: `print("Hello, World")`. No `main` function, no imports, no class wrapper, no type annotations.
- A file with no explicit entry point runs top-to-bottom as a script. Adding `fn main()` opts into structured-program mode.
- Type annotations are optional for local variables (inferred) but required on public function signatures (documentation and API stability).
- Generics, macros, and unsafe blocks each have distinct syntax that signals "you are now in advanced territory."
- The standard library API avoids generics in its most common entry points. `http.get(url)` returns a `Response`, not a `Result<Response<Body>, Error>`.

**Counter-examples**:
- **Java**: `public static void main(String[] args)` requires understanding access modifiers, static methods, return types, arrays, and string types — before printing a single line.
- **Rust**: The simplest program requires `fn main() {}`, and the simplest I/O requires understanding `Result`, `unwrap()`, and the `use` system. Reading a file involves `std::fs::read_to_string("file.txt").unwrap()` — four concepts (modules, paths, Result, unwrap) for one operation.
- **C++**: `#include <iostream>` and `std::cout << "Hello" << std::endl;` exposes the preprocessor, namespaces, operator overloading, and stream objects in line one.
- **Haskell**: `IO` monad is required for any program that does anything, but understanding monads is a months-long journey for most developers.

### 5. Tooling Is the Language

**Statement**: The formatter, package manager, test runner, and language server are first-class deliverables — not ecosystem afterthoughts.

**Rationale**: A modern programming language is not just a grammar and a compiler. Developers spend more time interacting with tooling (autocomplete, error squiggles, formatting, dependency management) than they spend reading the language specification. If these tools are third-party, fragmented, or inconsistent, the developer experience fractures regardless of how elegant the syntax is. Shipping unified tooling eliminates the "which formatter?" and "which build tool?" debates that plague ecosystems like JavaScript and C++.

**Corollary**:
- `lang fmt` formats all code to a single canonical style. No configuration file. No options. One style.
- `lang test` discovers and runs tests using a built-in test syntax (e.g., `test "math works" { assert(1 + 1 == 2) }`). No test framework to choose.
- `lang pkg` manages dependencies with a lockfile, reproducible builds, and a central registry — all shipped with the compiler.
- The language server (LSP implementation) is maintained by the core team and supports go-to-definition, rename, inline errors, and autocomplete from day one.
- `lang doc` generates documentation from code comments with zero configuration.

**Counter-examples**:
- **JavaScript**: Prettier vs. ESLint vs. Biome, npm vs. yarn vs. pnpm, Jest vs. Vitest vs. Mocha — every project starts with 30 minutes of tooling decisions before a single line of application code.
- **C++**: CMake, Make, Bazel, Meson, Ninja — the build system landscape is so fragmented that "how do I build this?" is a legitimate question for every new project.
- **Python**: `pip` vs. `poetry` vs. `pipenv` vs. `uv`, `venv` vs. `conda` vs. `pyenv` — environment management is the single largest source of frustration for new Python developers.
- **Rust** (partial counter): Rust's `cargo` is a gold standard for package management, but the formatter (`rustfmt`) and linter (`clippy`) shipped separately and took years to stabilize. This is the model to follow and improve upon.

### 6. Explicit Defaults, Implicit Escape Hatches

**Statement**: The common path should be explicit and readable; power-user shortcuts should exist but require a deliberate opt-in.

**Rationale**: "Explicit is better than implicit" is incomplete advice — taken literally, it produces Java-style boilerplate where every type is annotated, every exception is checked, and every allocation is manual. The real insight is that *defaults* should be explicit (you can read what happens without knowing the framework's conventions), but *escape hatches* should be available without ceremony when you need them. The language should make the safe, readable choice the path of least resistance, while never trapping experts in a sandbox.

**Corollary**:
- Variables are immutable by default. `let x = 5` is immutable; `let mut x = 5` opts into mutability. The common case (immutability) requires less typing.
- Functions are pure by default in the type system. Side effects are tracked but not with monadic syntax — a simple `!` suffix (e.g., `fn save!(record)`) marks effectful functions.
- Null does not exist in the default type system. Optional values use `Option<T>` with a concise `?` operator for propagation. You never get a null pointer error unless you explicitly enter an FFI boundary.
- Unsafe memory operations are available via `unsafe { }` blocks, but standard code never needs them.
- Dynamic typing is available via an `any` type for scripting-style prototyping, but static types are the default.

**Counter-examples**:
- **Java**: Checked exceptions force every caller to either handle or re-declare exceptions, even when 95% of handlers are `catch (Exception e) { throw new RuntimeException(e); }`. The "explicit" choice became ceremony.
- **Go**: `err != nil` after every function call is explicit but so repetitive that developers write `must()` wrappers to ignore errors — undermining the principle it was designed to uphold.
- **Python**: Everything is mutable and dynamic by default. Immutability and type safety require opt-in (`frozen=True` on dataclasses, `mypy` as an external tool). The default path is the unsafe path.
- **C**: No default safety for anything — buffer overflows, null dereferences, and undefined behavior are all the default. Every safety mechanism is opt-in and manual.

### 7. Fast from Edit to Answer

**Statement**: The time from saving a file to seeing results — or seeing an error — must feel instantaneous; sub-second feedback is a design constraint on the compiler.

**Rationale**: Cognitive flow depends on tight feedback loops. When a developer types a line, saves, and waits 30 seconds for a compiler, their mental model of the code has already drifted. Sub-second feedback keeps developers in the zone. This is not merely a performance goal — it is a *design constraint* that actively shapes language features: if a feature makes incremental compilation impossible, that feature must be redesigned. The compiler's speed is a user-facing feature, not an implementation detail.

**Corollary**:
- Incremental compilation is a first-class architecture decision. The compiler is designed around it from day one, not retrofitted.
- Type inference is bounded: the compiler never infers types across module boundaries, ensuring that changes to one file do not trigger re-checking of distant files.
- The REPL ships with the language and supports loading project files for interactive exploration.
- `lang run file.lang` executes a single file with no build step, no manifest, no project scaffolding — like `python file.py` or `go run file.go`.
- CI/CD builds of a medium-sized project (50k lines) should complete in under 10 seconds from a warm cache.

**Counter-examples**:
- **Rust**: Clean builds of medium projects routinely take 2-5 minutes. Incremental builds are faster but still measured in seconds-to-minutes. The borrow checker and monomorphization are architecturally expensive.
- **C++**: Full rebuilds of large projects can take hours. Header-file inclusion means changing one `.h` file can trigger recompilation of thousands of translation units.
- **Scala**: Compiler speed has been a persistent complaint. The sbt build tool's "warming up" phase adds seconds of latency to every edit-compile cycle.
- **TypeScript**: Large projects with complex generics and conditional types can produce multi-second type-checking pauses in the editor, breaking flow. The `tsc` compiler does not support true incremental checking across project references without manual configuration.

## Process Notes

### Step 1: Seed Discussion Summary

The PM framed the target as **professional developers who want to ship reliable software without fighting the language** — web backends, CLI tools, data pipelines, and general application development. The Architect surveyed prior art (Python's readability, Go's simplicity and tooling, Rust's safety model, TypeScript's gradual typing) and proposed 10 candidate principles: Readable by Default, One Obvious Way, Errors Are Values, Fast Feedback, Gradual Complexity, Batteries Included, Zero-Cost Clarity, Newcomer-Friendly by Design, Explicit Over Implicit, and Composition Over Inheritance.

### Step 2: Chaos Review Summary

The Chaos Agent challenged each candidate with adversarial scenarios:
- **"Readable by Default"** breaks down in math-heavy domains where symbolic operators *are* the readable form. Refined to "Readable at a Glance" with domain-specific escape hatches.
- **"One Obvious Way"** conflicts with concurrency, where channels and async/await serve genuinely different use cases. Resolved by designating one *default* model with lower-level alternatives in explicit modules.
- **"Errors Are Values"** punishes prototyping speed (every call needs handling). Resolved by making error propagation lightweight (`?` operator) and allowing a script-mode `!` that crashes on error.
- **"Zero-Cost Clarity"** was challenged as borrowing Rust's framing without Rust's mission. Dropped as a standalone principle; its concerns folded into "Explicit Defaults."
- **"Explicit Over Implicit"** was challenged with the Java boilerplate counter-argument. Reframed as "Explicit Defaults, Implicit Escape Hatches" to capture the nuance.
- **"Newcomer-Friendly"** was challenged as potentially patronizing to experts. Reframed as "Progressive Disclosure" — a structural commitment, not a marketing claim.

### Step 3: User Lens Summary

The New User agent ranked principles by personal impact:
1. **Errors That Teach** — "This is the single biggest thing. I've mass-quit languages because the errors were useless."
2. **Fast from Edit to Answer** — "If I can't see my change in under a second, I lose focus."
3. **One Obvious Path** — "I don't want to research five libraries before writing line one."
4. **Progressive Disclosure** — "If Hello World needs a class, I'm already gone."
5. **Tooling Is the Language** — "I shouldn't need a blog post to set up my editor."

The New User flagged two missing concerns: (a) deployment simplicity (single binary output), addressed by folding it into the Tooling corollary; and (b) community and documentation quality, acknowledged as important but out of scope for a *language design* principles document.

### Step 4: Evangelist Lens Summary

The Evangelist identified the strongest narrative anchors as **Errors That Teach** (unique differentiator — no mainstream language commits to this at the specification level), **One Obvious Path** (clear contrast with JavaScript/Python fragmentation), and **Tooling Is the Language** (the Go/Rust playbook, extended). "Progressive Disclosure" was praised as a structural commitment that backs up the tagline.

The Evangelist flagged "Zero-Cost Clarity" and "Composition Over Inheritance" as too implementation-focused for the principles document — they belong in a language design guide, not a manifesto. Both were cut.

The final tagline — **"Simple to start, powerful to stay."** — was chosen for capturing the progressive-disclosure philosophy in six words.

### Step 5: Synthesis Summary

The Architect and Spec Writer consolidated the 10 candidates into 7 final principles by: merging "Readable by Default" and "Newcomer-Friendly" into **Readable at a Glance** and **Progressive Disclosure** (two distinct concerns that were previously tangled); dropping "Zero-Cost Clarity" and "Composition Over Inheritance" as implementation-level concerns; reframing "Explicit Over Implicit" as the more nuanced **Explicit Defaults, Implicit Escape Hatches**; elevating error messages from a sub-point of "Fast Feedback" to a standalone principle (**Errors That Teach**); and renaming "Batteries Included" to **Tooling Is the Language** to emphasize that tooling is a *deliverable*, not a feature list.

### Step 6: PM Review

The PM performed final checks:
- **Redundancy**: "Readable at a Glance" and "Explicit Defaults" share surface area but serve different purposes (reading vs. writing code). Kept as distinct principles with a cross-reference.
- **Contradictions**: "One Obvious Path" and "Implicit Escape Hatches" could appear contradictory. Resolved by scoping: One Obvious Path governs the *default* experience; escape hatches are explicitly marked deviations.
- **Memorability**: All seven principles pass the "hallway test" — each can be explained in one sentence to a non-technical stakeholder.
- **Consensus vote**: PM ✅, Architect ✅, New User ✅, Evangelist ✅, Chaos Agent ✅ (with a standing objection that Principle 4's script-mode may create a "two languages" problem — noted and accepted as a known tension), Spec Writer ✅.

**Result: Consensus reached. Seven principles committed.**
